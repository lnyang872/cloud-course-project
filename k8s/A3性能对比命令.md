## 3. CloudShell 创建 executor=1 的 SparkApplication

```bash
cat > a3-spark-exec1.yaml <<'EOF'
apiVersion: sparkoperator.k8s.io/v1beta2
kind: SparkApplication
metadata:
  name: a3-spark-groupby-exec1
  namespace: default
spec:
  type: Python
  mode: cluster
  image: swr.cn-south-1.myhuaweicloud.com/cloud-course/a3-spark:v1
  imagePullPolicy: Always
  mainApplicationFile: local:///opt/spark/app/a3_spark_groupby.py
  sparkVersion: "3.4.3"
  restartPolicy:
    type: Never
  driver:
    cores: 1
    coreLimit: "1200m"
    memory: "1g"
    labels:
      version: "3.4.3"
    serviceAccount: spark
  executor:
    cores: 1
    instances: 1
    memory: "1g"
    labels:
      version: "3.4.3"
EOF
```

提交：

```bash
kubectl delete sparkapplication a3-spark-groupby-exec1 --ignore-not-found=true
kubectl delete pod a3-spark-groupby-exec1-driver --ignore-not-found=true
kubectl apply -f a3-spark-exec1.yaml
kubectl get pods -n default -w
```

完成后查看日志：

```bash
kubectl logs a3-spark-groupby-exec1-driver -n default | grep -v " INFO " | grep -v " WARN "
```

记录：

```text
PYSPARK_TIME_SECONDS=...
```

## 5. 计算加速比

假设记录时间如下：

```text
Pandas 时间 = Tp
PySpark executor=1 时间 = Ts1
PySpark executor=2 时间 = Ts2
```

计算：

```text
PySpark 1 executor 相对 Pandas 加速比 = Tp / Ts1
PySpark 2 executor 相对 Pandas 加速比 = Tp / Ts2
PySpark 2 executor 相对 1 executor 加速比 = Ts1 / Ts2
```

## 6. Amdahl 分析模板

```text
本实验选择电影类型 GROUP BY 聚合查询进行性能对比。Pandas 在单机环境中运行，适合中小规模数据，避免了分布式调度开销；PySpark 在 Kubernetes 集群中运行，虽然具备并行处理能力，但启动 Driver、申请 Executor、读取镜像和任务调度都会引入额外开销。根据 Amdahl 定律，程序中只有可并行部分才能通过增加 executor 获得加速，而数据读取、任务调度、结果收集和部分聚合阶段存在串行开销，因此 executor 从 1 增加到 2 时加速比通常达不到线性 2 倍。对于当前约几十 MB 的数据集，数据规模偏小，Spark 的分布式通信和调度成本占比较高，因此 PySpark 可能并不一定快于 Pandas。随着数据规模增大，可并行计算部分比例提高，PySpark 的优势会更加明显。
```
