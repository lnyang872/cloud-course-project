## 任务 1：应用容器化（10 分）

要求：

1. 基于 Dockerfile 模板完成镜像构建和推送。
2. 修改 `Dockerfile.backend`：
   - 保留多阶段构建结构；
   - 在 `requirements.txt` 中加入至少 1 个自选 Python 包，例如 `pandas`、`requests` 等。
3. 修改 `Dockerfile.frontend`：
   - 在 Nginx 默认首页 `index.html` 中加入本人学号和姓名。
4. 本地或华为云 ECS 运行：

```bash
docker compose up
```

5. 截图验证前后端通信正常，后端日志应显示收到请求。
6. 将两个镜像推送到 SWR，并截图保存 SWR 控制台镜像列表页面。

SWR 推送命令示例：

```bash
docker login -u cn-north-4@<AK> -p <SK> swr.cn-north-4.myhuaweicloud.com
docker tag backend:v1 swr.cn-north-4.myhuaweicloud.com/<ORG>/backend:v1
docker push swr.cn-north-4.myhuaweicloud.com/<ORG>/backend:v1
```

验收材料：

## 任务 2：CCE 集群搭建（8 分）

要求：

```bash
kubectl get nodes -o wide
```

验收要求：

- 所有 Worker 节点 `Status` 为 `Ready`；
- 截图包含 `VERSION` 列；
- Kubernetes 版本不低于 1.27。

## 任务 3：应用部署（12 分）

需要将任务 1 的镜像部署到 CCE 集群并对外暴露。

需要完成的 Kubernetes 资源：

| 资源 | 说明 | 关键要求 |
|---|---|---|
| Backend Deployment | 运行 Flask API | 副本数为 2；设置 `resources.requests` 和 `resources.limits`；通过 ConfigMap 注入 Redis 地址；通过 Secret 注入 Redis 密码 |
| Redis Deployment | 运行 Redis | 副本数为 1；`limits.memory <= 512Mi` |
| Backend Service | 对外暴露后端 | 类型为 `LoadBalancer`；添加 `kubernetes.io/elb.class: union` 注解 |
| Redis Service | 集群内访问 Redis | 类型为 `ClusterIP` |
| ConfigMap | 非敏感配置 | `REDIS_HOST=redis-svc`，`REDIS_PORT=6379` |
| Secret | 敏感配置 | Redis 密码使用 base64 编码后写入 `data` 字段 |

Secret 编码命令：

```bash
echo -n "your_password" | base64
```

验收要求：

```bash
kubectl get pods
```

所有 Pod 应为 `Running`。

访问接口：

```bash
curl http://<ELB_IP>/api/ping
```

返回结果应为：

```json
{"status":"ok"}
```

## 任务 4：持久化存储（10 分）

目标：为 Redis 配置 PVC，保证 Pod 重建后数据不丢失。

要求：

1. 创建 PVC。
2. `storageClassName` 选择华为云 `csi-disk`。
3. 修改 Redis Deployment，将 `/data` 挂载到该 PVC。
4. 验证 Redis 数据持久化。

验证流程：

```bash
kubectl get pvc
```

PVC 状态应为 `Bound`。

写入测试数据：

```bash
redis-cli SET testkey "hello"
```

删除 Redis Pod：

```bash
kubectl delete pod <redis-pod-name>
```

重建后查询：

```bash
redis-cli GET testkey
```

应返回：

```text
hello
```

验收材料：

- `kubectl get pvc` 截图，状态为 `Bound`；
- 写入数据截图；
- 删除 Pod 截图；
- 重建后查询数据仍存在的截图。

## 任务 5：ConfigMap Volume 挂载（5 分）

目标：将 Nginx 反向代理配置改为通过 ConfigMap Volume 挂载。

要求：

1. 新建 ConfigMap，`data` 中包含完整 `nginx.conf` 内容。
2. 修改前端 Deployment。
3. 将 ConfigMap 挂载到：

```text
/etc/nginx/conf.d/default.conf
```

4. 修改 ConfigMap 中后端端口值，例如 `5000 -> 5001`。
5. 执行：

```bash
kubectl apply -f <configmap-yaml>
```

6. 进入前端 Pod 验证配置文件已更新：

```bash
kubectl exec -it <frontend-pod-name> -- cat /etc/nginx/conf.d/default.conf
```

报告中需要说明：

- Volume 挂载适合配置文件类内容；
- `envFrom` 适合环境变量类配置；
- 两者更新方式和应用感知方式不同。

## 任务 6：HPA 弹性伸缩（5 分）

目标：为后端 Deployment 配置 HPA，并通过压测证明弹性伸缩生效。

要求：

1. 确认 metrics-server 可用：

```bash
kubectl top nodes
```

2. 创建 HPA：

| 参数 | 值 |
|---|---|
| minReplicas | 1 |
| maxReplicas | 4 |
| targetCPUUtilizationPercentage | 60 |

3. 使用 `ab` 压测：

```bash
ab -n 10000 -c 200 http://<ELB_IP>/api/ping
```

4. 观察 Pod 扩容：

```bash
kubectl get pods -w
```

验收材料：

- Pod 数量从 1 增加到 2 或更多的截图；
- 停止压测后 Pod 缩回 1 的截图；
- 报告中分析扩容延迟、冷却时间和 HPA 降本价值。

分析要点：

- 扩容延迟来自 metrics 采集周期和 HPA 评估间隔；
- 冷却时间用于避免频繁扩缩容抖动；
- HPA 可以在低负载时减少副本数，从而节省资源成本。

### B-1：并行算法实现（10 分）

三选一实现串行版和 MPI 并行版。

| 题目 | 串行算法要点 | MPI 通信要点 |
|---|---|---|
| 并行矩阵乘法 | 三重循环，复杂度 O(n³) | Scatter 分发行块，Gather 收集结果 |
| 数值积分（梯形法） | 均分区间累加 | Scatter 子区间，Reduce 求和 |
| 并行排序（奇偶换序） | 冒泡排序 | 相邻进程比较与交换 |

要求：

- 代码中对每个 MPI 通信原语添加注释说明数据流向；
- 并行版结果与串行版一致；
- 截图对比输出结果；
- 报告附通信模式示意图，手绘拍照也可。

---

### B-2：性能测试与 Amdahl 分析（15 分）

要求：

1. 固定问题规模，例如矩阵 `N=800` 或积分区间 `10^7` 个点。
2. 在 1、2、4 个 MPI 进程下各运行 3 次取平均。
3. 填写性能表格。
4. 绘制“实测加速比 vs Amdahl 理论加速比”双折线图。
5. 根据实测数据估算可并行比例 `f`。
6. 分析实测与理论差距。

性能表模板：

| 进程数 p | 平均运行时间 T(p) / s | 实测加速比 S | Amdahl 理论值 |
|---:|---:|---:|---:|
| 1 |  | 1.00 | 1.00 |
| 2 |  |  |  |
| 4 |  |  |  |

分析角度：

- 通信开销；
- 进程同步；
- 负载不均衡；
- 网络延迟；
- 数据划分开销。

---

### B-3：非阻塞通信优化（5 分）

要求：

1. 将 B-1 中至少一处关键通信改为非阻塞模式：
   - `comm.Isend`
   - `comm.Irecv`
2. 实现计算与通信重叠。
3. 对比 4 进程下阻塞版与非阻塞版执行时间。
4. 报告中分析非阻塞通信何时有效、何时改善有限。

分析角度：

- 当计算量大、通信可隐藏时，非阻塞通信更有效；
- 当任务通信占比低或计算量太小，优化效果有限；
- 网络延迟、消息大小、进程同步都会影响效果。
